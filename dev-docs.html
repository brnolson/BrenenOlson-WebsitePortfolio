Tethr — Wedding Planning Suite

Tethr is a multi-product wedding planning platform. Couples get a suite of connected tools — from digital save-the-dates to AI-powered seating charts — all sharing a single guest list and account. Planners get their own dashboard to manage multiple client events.

ℹ️ All six products (Save the Date, RSVP, TableTag, Website, Maestro, Guest List) share the same Supabase backend and Vue frontend. You're working in a monorepo-style single SPA, not separate services.

Products at a glance

Save the Date RSVP / Invites TableTag (Seating) Wedding Website Maestro (Messaging) Guest List Guest Concierge (AI) Planner Dashboard
// getting started

Local Setup

You need Node 20+, a Supabase account (ask Brenen for credentials), and a Stripe test key.

1. Clone & install

git clone https://github.com/brnolson/tethr-wedding-webapp
cd tethr-wedding-webapp
npm install

2. Environment variables

Copy .env.example to .env.local and fill in the values Brenen shares:

VITE_SUPABASE_URL=https://<project>.supabase.co
VITE_SUPABASE_KEY=eyJ...
VITE_REDIRECT_URL=http://localhost:5173
 
# For running tests only:
SUPABASE_SERVICE_ROLE_KEY=eyJ...
PW_COUPLE_EMAIL=pw-couple-1@tethr-tests.local
PW_COUPLE_PASSWORD=...

3. Dev server

npm run dev # starts at http://localhost:5173
npm run build # production build → dist/
npm run lint # ESLint with auto-fix
npm run test:e2e # Playwright tests (needs .env.local)
npm run test:e2e:ui # Playwright interactive UI
⚠️ The test suite hits a real Supabase database and auto-creates test accounts on first run via tests/support/global-setup.js. Don't run it against the production project.

Routing & Auth

All routes are defined in src/router/index.js. Protected routes carry meta: { requiresAuth: true } and redirect to /login if the user is not authenticated.

Key route patterns

Route types
/ Public marketing landing page
/save/dashboard Protected dashboard — user manages their save-the-dates here
/save/:id Public guest-facing view — no auth required
/view/:slug Public wedding website or table-tag seating view
/invite/:token Collaboration invite acceptance flow
/connect/* Planner ↔ couple connection flows

User & Profile state

The root App.vue fetches the Supabase auth session and user profile on mount, then provides both via Vue's provide/inject so any child component can access them. Planners have an active_wedding_user_id field on their profile that resolves which couple's data to show — the useEffectiveWedding composable handles this transparently.

// data layer

Database & Edge Functions

Supabase PostgreSQL is the single source of truth. Row-level security (RLS) policies enforce multi-tenant access — users can only read their own rows.

Core tables

Tables (simplified)
user_profilesPlan type, user type (couple/planner), active wedding context
save_the_datesSTD events — owns collaborator + planner connection columns
rsvp_eventsRSVP events, form fields, template config
tabletag_eventsSeating events, tables, guests, send status
master_guestsShared guest directory, linked to all products
maestro_messagesScheduled messages with status + retry logic
wedding_websiteWebsite config, sections, published slug
concierge_configsAI chatbot settings per couple

Edge Functions

Serverless Deno functions in supabase/functions/ handle anything that needs a secret key or heavy computation. They're called from the frontend via supabase.functions.invoke().

Key Edge Functions
send-table-tagsSends table tag emails + SMS to all guests in an event
maestro-scheduled-sendCron-invoked job that dispatches pending scheduled messages
guest-conciergeProxies user messages to Claude API, returns AI reply
ai-seat-guestsClaude-powered seating algorithm with guest constraints
stripe-checkoutCreates Stripe checkout sessions for plan upgrades
stripe-webhookHandles Stripe events (payment success, cancellation, etc.)
send-account-emailAll transactional emails — invites, confirmations, nudges
import-guests-to-masterImports guests from any product into the master guest list
💡 Schema changes always go through supabase/migrations/ — never edit the database directly. New migrations are named YYYYMMDDHHMMSS_description.sql.

Testing Suite

Tethr uses Playwright for all testing — there are no unit tests. Every spec hits a real Supabase test project and a real browser, so what passes actually works.

Test projects (browser contexts)

Project Auth State Used For
anonymous-chromium Not logged in Public pages, marketing, guest-facing views
couple-chromium Couple account All product dashboards and CRUD flows
planner-chromium Planner account Planner dashboard, client management
sharing-chromium Couple + Couple2 Collaboration invite/accept flows
mobile-chromium Couple account Responsive behavior at 390×844

Running tests

npm run test:e2e # all tests, headless
npm run test:e2e:ui # Playwright interactive UI
npm run test:e2e:headed # headed, single worker
npx playwright test tests/e2e/rsvp-crud.spec.js # run one file
npm run test:e2e:report # open last HTML report

Test structure

Each spec file is independent — they don't share state between files. Test accounts are auto-created on first run by tests/support/global-setup.js and auth sessions are saved to tests/.auth/ so login only happens once per run.

Coverage areas

CRUD for all 6 products Auth flows Onboarding Collaboration / sharing Public guest pages Planner ↔ couple connection Responsive / mobile Smoke tests (marketing pages)
⚠️ Tests are configured with 90s timeout and 0 retries locally (2 retries in CI). If a test is flaky, investigate rather than bumping the retry count — flakiness usually means a missing await expect(...).toBeVisible() before an action.
// shipping

Deployment

Frontend deploys automatically to Vercel on every push to main. Edge functions deploy via the Supabase CLI.

Frontend (Vercel)

Vercel rewrites all paths to /index.html (configured in vercel.json) so Vue Router handles client-side navigation. No server-side rendering — it's a pure SPA.

Edge Functions (Supabase CLI)

supabase functions deploy send-table-tags
supabase functions deploy maestro-scheduled-send
supabase functions deploy --all # deploy everything

Database migrations

supabase migration new add_column_to_table # creates a new migration file
supabase db push # applies migrations to remote
ℹ️ Production secrets (Stripe keys, Resend API key, Telnyx key) live in Vercel environment variables and Supabase vault. You won't have these locally — ask Brenen for test-environment equivalents.
